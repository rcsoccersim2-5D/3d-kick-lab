# Ball & Kick Physics — 3-Way Comparison

Comparing three RoboCup-family simulators' ball/kick physics:

- **rcssserver** — the official 2D league server (`D:\workspace\robo\ss2d\rcssserver`)
- **SimSpark** — the official 3D league server, ODE-based rigid-body physics (`D:\workspace\robo\ss2d\SimSpark`)
- **3d-kick-lab** — this repo, a browser prototype extending rcssserver's 2D model with a height/z axis (`D:\workspace\robo\ss2d\3d-kick-lab`)

Because a plain 1-column-per-repo table hides interesting nuances, several rows are split into **scenario variants**:

- SimSpark is shown twice: **(a) as-is** (kicks can target any point/area of the ball via a 3D direction vector), and **(b) "flattened"** — an imagined config where the kick's vertical angle (`phi`) is hard-locked to `0°`, which collapses it to a 2D-equivalent, no-contact-area kick model like rcssserver. This isolates *what height/contact-point support actually buys you* in SimSpark's formula.
- 3d-kick-lab is shown twice: **`precise_bounce_timing = OFF`** vs **`= ON` (default since the update below)**, since this single flag changes the ground-collision-timing model from discrete/naive to continuous/analytical. (The default flipped from `OFF` to `ON` after this comparison showed rcssserver's own post-collision code already matches the `ON` style — see §4/§5.)

All file:line references were captured by direct source inspection (read-only) on the current checkout.

## 1. Kick Formulation — On the Ground

| | rcssserver (2D) | SimSpark (3D, as-is) | SimSpark (3D, flattened — `phi≡0`) | 3d-kick-lab (`precise_bounce_timing` ON/OFF — irrelevant here, same in both) |
|---|---|---|---|---|
| Command shape | `(kick power dir)` — 2 scalars | `(kick power angle)` where `angle`≈`phi` (latitudinal, 0–50°) + implicit `theta` (horizontal, computed from body→ball vector) | Same command, but `phi` forced to `0°` | `kick(power, dirDeg, loftDeg)` — 3 scalars (adds explicit `loft`) |
| Contact-point / kick-area on ball | **None** — ball is a point; only distance+angle-to-ball-center matter | **Implicit** — `theta = atan2(dy,dx)` is literally the direction from *player origin* to *ball center*, so kicking "around" the ball naturally changes horizontal direction; `phi` (elevation) sets how "under" the ball the foot strikes, giving vertical lift. Not a true surface-contact-point/spin model — no spin/Magnus. | **None** — with `phi=0` the force is always horizontal, functionally identical to rcssserver: ball area/height is irrelevant to the kick outcome | **None** — ball treated as dimensionless point; `ball_size` explicitly excluded from flight math (only used for kickable-distance check) |
| Effective power formula | `eff = power·kick_power_rate·(1 − 0.25·dirDiff/π − 0.25·distBall/kickable_margin)` | `force = (cosθ·sinφ, sinθ·sinφ, cosφ) · forceFactor(4.0) · clamp(power,1,maxPower)` — no dirDiff/distBall penalty term at all (validity is pass/fail, not graded) | Same as SimSpark as-is but `sinφ=0,cosφ=1` → pure horizontal unit vector · forceFactor · power | `eff = power·kick_power_rate·(1 − 0.25·dirDiff/π − 0.25·distBall/kickable_margin − heightCost·heightFrac)` then `·(1 − loftCost·loft/(π/2))` — a strict superset of rcssserver's formula |
| Validity/kickable check | 3-part *graded* efficiency penalty (never hard-fails except if ball outside `kickable_margin` circle) — 2D distance only | Hard boolean gate: reject if `agent.z > playerRadius+0.01` OR **3D** distance `> playerRadius+ballRadius+margin` | Same gate, but since kicks never add z-velocity, "reachability" degenerates to the 2D case in practice (agent/ball are always near z≈0 in this mode) | Boolean gate: 2D kickable circle **AND** `ball.z ≤ player_reach_height` |
| Noise model | `kick_noise = polar(rand(0,maxRand), rand(-π,π))`, `maxRand` scales with `pos_rate`+`speed_rate` | Optional Gaussian noise on force/theta/phi (`SetNoiseParams`, off unless configured) | Same optional noise, but phi-noise moot since phi is clamped to 0 | Same conceptual formula as rcssserver (`ball_rand`), default **0** (disabled) — off by default unlike rcssserver's `kick_rand=0.1` |
| Result | Instant acceleration impulse, consumed in next integration step | **Sustained** force+torque applied over `mSteps=10` physics ticks (~0.2s), not instantaneous | Same sustained-force mechanism, purely horizontal | Instant acceleration impulse (single step), like rcssserver — no multi-tick force sustain |

**Takeaway:** rcssserver and "flattened SimSpark" converge to functionally the same 2D kick model (graded efficiency vs boolean gate is the main structural difference). Full SimSpark's only "kick-area" concept is the *elevation angle* `phi`, not a literal point-of-contact on the ball surface — so even the "3D, as-is" column doesn't support true off-center/spin kicks. **3d-kick-lab is the only one of the three that explicitly models a height-based power cost (`height_power_cost`) and a loft-power trade-off (`loft_power_cost`)** — neither rcssserver nor SimSpark penalizes power for kicking a ball at height or kicking it upward.

## 2. Kick Formulation — In the Air (imparting height/loft)

| | rcssserver (2D) | SimSpark (3D, as-is) | SimSpark (3D, flattened) | 3d-kick-lab (either bounce mode) |
|---|---|---|---|---|
| Can a kick add vertical velocity? | **No** — no z-axis exists at all | **Yes** — `phi ∈ [minAngle, maxAngle]` (default `0°–50°`) tilts the force vector: `force.z = cosφ` | **No** (phi forced to 0 — by definition of this variant) | **Yes** — explicit `loftDeg` parameter (0–90°), `accelZ = effPowerTotal·sin(loft)` |
| Max elevation angle | N/A | 50° (`SetAngleRange` default) | 0° | Unbounded 0–90° (UI-controlled) |
| Power trade-off for lofting | N/A | None found — `phi` only redirects existing force vector, no extra power cost | N/A | Explicit: `effPowerTotal = effPower·(1 − loft_power_cost·loft/(π/2))`, `loft_power_cost=0.4` |
| Kicking a ball that's already airborne | **N/A** (ball never leaves ground) | **Yes, by design** — kick validity uses **3D** distance (not 2D/horizontal), so a grounded player can strike an elevated ball (volley) as long as it's within reach-sphere | **N/A in practice** — same 3D-distance validity check exists in code, but since nothing ever gives the ball height in this variant, it never arises | **Yes** — gated by `ball.z ≤ player_reach_height` (default 2.0) rather than a proximity-sphere; this is a *height ceiling*, not a *3D distance* check — structurally different from SimSpark's approach |

**Takeaway:** SimSpark and 3d-kick-lab both support lofted/aerial kicks, but via different validity philosophies — SimSpark checks true 3D proximity (sum-of-radii sphere), while 3d-kick-lab checks 2D proximity **plus** a separate height ceiling. rcssserver has no equivalent at all (this is the fundamental 2D vs 3D divide). Only 3d-kick-lab charges extra power for high/lofted kicks.

## 3. Ball Movement on the Ground (rolling/decay)

| | rcssserver (2D) | SimSpark (3D) | 3d-kick-lab (both bounce modes — identical here) |
|---|---|---|---|
| Model type | Exponential geometric decay: `v ← v·decay` every cycle | Force-based linear viscous damping every tick: `F = −v·linearDrag` fed into the ODE integrator, plus real Coulomb-like friction (`mu`) from ODE contact joints | Same exponential-decay style as rcssserver: `v ← v·ball_decay` while grounded |
| "Friction" constant name & value | `ball_decay = 0.94` ([serverparam.cpp:152](D:/workspace/robo/ss2d/rcssserver/src/serverparam.cpp:152)) | `linearDrag = 0.01` (ball, [ball.rsg:23](D:/workspace/robo/ss2d/SimSpark/rcssserver3d/data/rsg/agent/ball.rsg:23)) — a *different physical concept* (viscous drag), plus separately `mu=∞` default / mixed contact friction from ground surface | `ball_decay = 0.94` (same name & value as rcssserver, [physics.js:33](D:/workspace/robo/ss2d/3d-kick-lab/physics.js:33)) |
| Airborne damping | N/A (no air phase) | Same `DragController` mechanism, using `angularDrag=0.00005` for spin damping too | Separate constant `air_decay = 0.999` (near-frictionless while airborne) — **rcssserver has no equivalent since it has no air phase; SimSpark uses the same `linearDrag` for both air and ground (no separate "grounded" multiplier)** |
| Stop condition | None — asymptotic decay, velocity never truly reaches 0 | None explicit — asymptotic under drag+friction, further slowed by contact friction when resting on the plane | Explicit thresholds: `roll_stop_speed=0.05` (xy), `bounce_stop_speed=0.05` (z) — ball is hard-clamped to exactly 0 once below threshold |
| Positional noise while moving | `ball_rand=0.05` applied every cycle regardless of contact state | None found in ball motion (kick-time noise only, via `SetNoiseParams`) | `ball_rand` param exists (mirrors rcssserver name), **default 0** (disabled) |
| Max speed / accel clamp | `ball_speed_max=3.0`, `ball_accel_max=2.7` (hard clamps every cycle) | No explicit global "max ball speed" clamp found — governed only by physics (drag, forces) | Same names/values as rcssserver: `ball_speed_max=3.0`, `ball_accel_max=2.7` — but **only applied to the xy component**, z is unclamped |

**Takeaway:** rcssserver and 3d-kick-lab share the *exact same decay philosophy and even the same constant name/value* (`ball_decay=0.94`) — 3d-kick-lab is literally built as a superset of rcssserver's ground model. SimSpark is architecturally different: it's a real force/impulse-based rigid-body sim (ODE), so "decay" isn't one number but an interaction of `linearDrag` (viscous damping, tiny at `0.01`) and contact friction (`mu`, mixed per-surface) — there's no direct 1:1 constant mapping, only an analogous *role*.

## 4. Ball Hits the Ground (bounce / vertical collision)

| | rcssserver (2D) | SimSpark (3D) | 3d-kick-lab `precise_bounce_timing = OFF` (legacy/experimental) | 3d-kick-lab `precise_bounce_timing = ON` (default) |
|---|---|---|---|---|
| Applicable at all? | **No** — no vertical axis, ball never "falls" | Yes | Yes | Yes |
| Collision detection timing | N/A | **Discrete, per-step**: ODE's `dSpaceCollide`→`dCollide` tests geometry only at each step's *already-moved* position — no continuous/swept test. If the ball would tunnel past the ground within one 0.02s tick, no contact is generated that tick (theoretical risk only, not practically hit at RoboCup ball speeds) | **Discrete, post-step, snap-to-zero**: ball is integrated straight-line for the whole cycle; if `z` ends up `≤0`, position is hard-clamped to `z=0` and bounce velocity is computed from the raw incoming `vz` — the portion of the fall that happened *after* the true zero-crossing is silently discarded ([physics.js:354,361-379](D:/workspace/robo/ss2d/3d-kick-lab/physics.js:354)) | **Continuous/analytical**: computes the exact linear-interpolation fraction of the cycle where `z` crosses 0 (`frac = z0/(z0−newZ)`), reflects velocity at that instant, then re-integrates the *remaining* fraction of the cycle with real kinematics (`pos = v·t − ½g·t²`) so the ball never visibly dips below ground ([physics.js:305-356](D:/workspace/robo/ss2d/3d-kick-lab/physics.js:305)) |
| Penetration correction style | N/A | **Gradual / soft-constraint**: ODE's ERP/CFM bias only corrects a *fraction* of overlap per tick (ball-ground mixed `softERP≈0.35`, i.e. ~35%/tick) — takes several ticks to fully resolve, not instantaneous | **Instantaneous snap** — position forced to exactly `z=0` in the same step the overlap is detected (no gradual settling of *position*; only *velocity* decays over subsequent bounces) | **Instantaneous, but analytically exact** — no snap discontinuity at all; the crossing point itself is computed, so there's no error to gradually correct |
| Restitution / bounce coefficient | N/A | `bounce = 0.8` (ball-side, [ball.rsg:38](D:/workspace/robo/ss2d/SimSpark/rcssserver3d/data/rsg/agent/ball.rsg:38)), mixed with ground's (ground itself disables bounce mode, so ball's value wins) | `ball_bounce_restitution = 0.65` ([physics.js:51](D:/workspace/robo/ss2d/3d-kick-lab/physics.js:51)) — same *role* as SimSpark's `bounce`, different name & value | Same `ball_bounce_restitution = 0.65`, identical to OFF mode |
| Minimum-velocity-to-bounce threshold | N/A | `bounce_vel = 2.0` (min relative velocity for the *bounce* contact mode to trigger vs. a "soft landing") | `bounce_stop_speed = 0.05` — much smaller in scale (units differ) but same *role*: below this, treat as settled instead of bouncing again | Same `bounce_stop_speed = 0.05` |
| Normal→tangential impulse coupling (does a bounce also cost horizontal speed?) | N/A | **Yes, inherently** — ODE resolves the ball-ground contact as a single LCP solve where the tangential (friction) impulse is capped by `mu × normalImpulse` (Coulomb friction pyramid), so a harder vertical impact mechanically bleeds off more horizontal speed in that same solve, by construction | **New (`bounce_friction_mu`, default 0.3, added after this comparison)**: `_applyBounceFriction()` explicitly computes `normalImpulse = \|vz before\| − \|vz after\|` at each bounce and scales `vx,vy` down by up to `bounce_friction_mu × normalImpulse` — an explicit, hand-written approximation of the same idea, rather than an emergent property of a general constraint solver. `bounce_friction_mu=0` reproduces the pre-update behavior (no coupling at all) | Same `bounce_friction_mu` mechanism, identical in both timing modes (called from both branches' settle/bounce sub-cases) |
| Known artifact/bug | — | None documented in the code we reviewed | Micro-bounce risk explicitly documented and guarded against: `resting` flag skips re-applying gravity to an already-settled ball, since `gravity(0.15) > bounce_stop_speed(0.05)` would otherwise cause perpetual tiny bounces ([physics.js:282-292](D:/workspace/robo/ss2d/3d-kick-lab/physics.js:282)) | Same guard still applies (shared code path for the settled/resting case) |

**Takeaway:** This is the most direct 3-way (well, 4-way) parallel in the whole comparison, since the user's earlier question specifically probed it. **SimSpark's "soft/gradual correction over several ticks" and 3d-kick-lab's "OFF = snap-to-zero" are two different strategies for the same underlying discrete-detection problem** — SimSpark tolerates a small penetration and undoes it gradually via constraint solving, while 3d-kick-lab's OFF mode simply erases the error instantly (visually indistinguishable most of the time, but WILL differ if you look at exact height mid-bounce). 3d-kick-lab's **`precise_bounce_timing=ON`** is the only one of the three engines/modes that computes an actual **analytical time-of-impact** within the step — this is closer in spirit to true continuous collision detection than what SimSpark's ODE backend does (ODE is discrete+soft-corrected, not continuous, despite being the more "production" engine). **Update:** `precise_bounce_timing=ON` is now 3d-kick-lab's default (see §4/§5 finding that rcssserver's own post-collision code already matches this style), and a new `bounce_friction_mu` parameter closes the other gap identified in this comparison — previously the lab's bounce touched only `vz`, unlike SimSpark's inherently-coupled LCP solve; the ball's horizontal speed is now explicitly reduced in proportion to the vertical impulse absorbed at each bounce.

> **Important correction/addition**: rcssserver has **no ground-bounce concept at all** (no vertical axis, as noted), but it turns out rcssserver's *existing* **ball↔goal-post** collision code is itself a fully continuous/analytical time-of-impact scheme — i.e. structurally equivalent to `precise_bounce_timing=ON`, not `OFF`. See §5 below for the full trace. This means the closest "same style as `ON`" precedent for adding a real height/ground-bounce feature to rcssserver already exists in its own codebase (the post-collision solver), just applied to a different axis/geometry (2D circle boundary instead of a `z=0` plane).

## 5. Ball Hits a Goal Post

|---|---|---|---|

| Multiple bounces within one cycle | Yes — the `while` loop at [object.cpp:559-565](D:/workspace/robo/ss2d/rcssserver/src/object.cpp:559) repeats the exact-intersection + reflect + re-integrate sequence against the (possibly new) nearest post until the remaining path no longer crosses any post circle, so several bounces inside a single simulation cycle are each resolved analytically in sequence | Not specifically documented for posts; general ODE constraint solve handles simultaneous contacts within `dWorldStep`'s single LCP solve, but still per-tick, not per-sub-bounce | N/A |
| Restitution used | Implicit 100% (magnitude explicitly preserved — `vel.r()` unchanged) — **more "bouncy" than either 3D engine's ground contact** | Default `bounce≈0.8` (mixed defaults, both sides at 0.8) — **bouncier than SimSpark's tuned ground contact** (ground mixes down to `softERP=0.35`-style softness; post uses raw untuned ODE defaults) | N/A |
| Same code path as ball-vs-ground? | **No** — a distinct, special-cased, continuous-style reflection block, separate from the generic player/ball push-apart+`-0.1×` velocity kill used elsewhere (that other path — see §6 — IS the discrete/post-hoc style, i.e. rcssserver actually contains **both** styles, just for different collision types) | **Yes** — literally the same generic contact-joint code as ball-vs-ground; the only difference is which (default vs custom) surface parameters get mixed in | N/A (not implemented) |
| Deliberate tuning? | Yes — magnitude-preserving reflection was clearly a specific design choice | **No** — SimSpark's `.rsg` files have the goalpost `ContactJointHandler` block *explicitly commented out* ([soccer.rsg:113-115,181-183](D:/workspace/robo/ss2d/SimSpark/rcssserver3d/data/rsg/agent/soccer.rsg:113)), i.e. posts were deliberately left at engine defaults while the ground surface was hand-tuned (`softERP=0.2`, `softCFM=0.01`) | N/A |

**Takeaway (updated):** rcssserver's ball↔goal-post collision is architecturally a **continuous/analytical time-of-impact scheme** — equivalent in spirit to 3d-kick-lab's `precise_bounce_timing=ON`, not `OFF` — despite rcssserver being the oldest/2D of the three projects. This is a genuinely useful precedent: it shows the "exact crossing point + reflect + re-integrate remainder" pattern already exists and is proven inside rcssserver's own C++ codebase (`object.cpp`'s `intersect()`/`nearestPost()`), just applied to a 2D circle boundary rather than a `z=0` ground plane — a strong candidate blueprint if/when adding a real ground-bounce feature to rcssserver directly (as opposed to prototyping it in 3d-kick-lab first). SimSpark, by contrast, treats **ball-vs-post exactly the same as ball-vs-ground** (both go through the identical generic ODE contact-joint mechanism, discrete detection + gradual soft correction) — the only difference between SimSpark's post and ground contacts is which surface parameters happen to be configured vs left at defaults, not the underlying detection/correction algorithm. 3d-kick-lab has no goal-post physics at all yet (confirmed absent, per your instruction to skip this cell) — §7's overall-observations note suggests mimicking rcssserver's continuous approach would actually be a natural fit, since the lab's `ON` mode already uses the same underlying philosophy.

## 6. Ball Hits a Player

| | rcssserver (2D) | SimSpark (3D) | 3d-kick-lab |
|---|---|---|---|
| Is there a passive (non-kick) collision response at all? | **Yes** — real, if crude, response | **Implicit/generic** — same ODE contact-joint system would apply if agent and ball geometries overlap outside a kick (agent bodies are also ODE rigid bodies with colliders) | **No** — player is a static point/marker in the physics model; only used as a kickable-range reference, never a rigid collider |
| Detection | Squared-distance circle test: `dist² < (ball_size+player_size)²` ([stadium.cpp:1671](D:/workspace/robo/ss2d/rcssserver/src/stadium.cpp:1671)) | Generic ODE narrow-phase (`dCollide`) between ball sphere geom and agent body geoms | N/A — the only "distance" check that exists is the kickable-radius gate, which is a *permission* check, not a collision |
| Position resolution | Symmetric push-apart along midpoint line, iterated ≤10×/cycle ([stadium.cpp:1749-1825](D:/workspace/robo/ss2d/rcssserver/src/stadium.cpp:1749)) | Same generic contact-joint constraint solve used everywhere else in SimSpark (no dedicated player-collision code found; not deeply investigated this round) | N/A |
| Velocity resolution | **Both** objects: `vel *= −0.1` (crude bounce-back at 10% speed, angle-independent — [object.h:596](D:/workspace/robo/ss2d/rcssserver/src/object.h:596)) | Standard ODE restitution/friction mixing (agent + ball surface params), momentum consistent since it's a real rigid-body solve | N/A |
| Physical realism | Deliberately simplistic / arcade-like | Most physically consistent of the three (real constraint-based rigid body dynamics) | Not modeled |

**Takeaway:** 3d-kick-lab is the outlier here — since its player is a fixed reference point rather than a rigid body, there is **no ball-vs-player physical collision at all**, only the kickable-range permission gate. rcssserver has a deliberately crude, non-physical bounce-back (`-0.1×` velocity, direction-independent) — a hallmark of the 2D server's "good enough for gameplay" approach — while SimSpark, being a full rigid-body engine, resolves this the same way it resolves every other contact (generically, via ODE).

> **Cross-reference**: this ball-vs-player path is rcssserver's **discrete/post-hoc** collision style (move first, detect overlap on the resulting position, then correct) — i.e. the `precise_bounce_timing=OFF` equivalent. Combined with §5's finding that ball-vs-post is the **continuous/analytical** style (`ON` equivalent), rcssserver actually contains *both* of 3d-kick-lab's bounce-timing philosophies simultaneously, just applied to different collision types rather than exposed as a single toggle.

## Overall Observations

1. **3d-kick-lab is architecturally a direct extension of rcssserver**, not a re-implementation of SimSpark's approach — it reuses rcssserver's exact parameter names and values (`ball_decay`, `kick_power_rate`, `kickable_margin`, `ball_speed_max`, `ball_accel_max`) for the horizontal/xy plane, and only adds new, clearly-separated concepts for the vertical axis (`gravity`, `loft`, `air_decay`, `ball_bounce_restitution`, `bounce_friction_mu`, `height_power_cost`, `player_reach_height`). This makes it easy to "port back" into rcssserver's C++, matching the stated design intent in its file header. The newest of these, `bounce_friction_mu`, is explicitly inspired by SimSpark/ODE's coupled normal+tangential impulse resolution rather than by anything in rcssserver (which has no vertical axis at all).
2. **SimSpark is the only one of the three built on a general-purpose rigid-body engine (ODE)** — this buys real physical consistency (same contact-joint code handles ground, posts, and implicitly players) at the cost of "magic constant" tuning being scattered across many `.rsg`/`.rb` config files rather than one central params object, and at the cost of no built-in continuous collision detection (contrary to what its production-grade reputation might suggest).
3. **The `precise_bounce_timing` toggle in 3d-kick-lab is a deliberate teaching/comparison tool, and is now the lab's default (`true`/ON)**: this update flipped the default from `OFF` to `ON` after this very comparison showed that rcssserver's own goal-post collision code already uses the continuous/analytical `ON`-style scheme (see §4/§5) — making `ON` both the more accurate *and* the more server-authentic choice. `OFF` reproduces the "naive discrete" failure mode that *could* theoretically also affect SimSpark/ODE at extreme velocities, and remains available as a legacy/experimental A-B toggle — a good concrete illustration of the general discrete-vs-continuous collision detection trade-off discussed for SimSpark, but implemented in a much simpler/more explicit way (closed-form single-axis crossing) than a general-purpose sub-stepping/CCD algorithm would need to be.
4. **Goal-post handling is the sharpest architectural divergence**: rcssserver hand-codes a special elastic reflection, SimSpark reuses its one generic mechanism, and 3d-kick-lab has none yet — a natural "next feature" to prototype, and the comparison above (§5) suggests either mimicking rcssserver's simple magnitude-preserving reflection (fastest to implement, matches the lab's existing simplicity) or generalizing the lab's ground-bounce math to also apply to vertical/side box collisions (closer in spirit to SimSpark's generic-handler philosophy).
